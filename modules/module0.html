Module 0: Hoe Computers Echt Werken

Voordat je √©√©n regel C# schrijft, moet je begrijpen waar die code leeft.

1. De Von Neumann Architectuur

Elke computer die je ooit hebt aangeraakt (je laptop, je telefoon, je slimme koelkast) werkt volgens hetzelfde basisprincipe uit 1945: de Von Neumann Architectuur.

+---------------------------+          +-------------------------+
|           CPU             |          |        MEMORY           |
|  +---------------------+  |          |  (RAM)                  |
|  | Control Unit (CU)   |  | <======> |  +-------------------+  |
|  +---------------------+  |          |  | 0x00: Instructie  |  |
|  +---------------------+  |          |  | 0x01: Data        |  |
|  | Arithmetic Logic    |  |          |  | ...               |  |
|  | Unit (ALU)          |  |          |  | 0xFF: Stack/Heap  |  |
|  +---------------------+  |          |  +-------------------+  |
+---------------------------+          +-------------------------+
            ^                                      ^
            |          HUGE BOTTLENECK             |
            +======================================+
                        System Bus

Figuur 1: De CPU haalt data op uit RAM via de System Bus.

Er zijn drie hoofdrolspelers:

  1. CPU (The Brain): Kan rekenen (1+1) en beslissen (IF x > 0). Het is extreem snel, maar heeft bijna geen geheugen (alleen "Registers").
  2. RAM (The Workbench): Je werkgeheugen. Alles wat nu draait, leeft hier. C# objecten leven hier. Het is snel, maar vluchtig (weg als stroom uitvalt).
  3. Storage (The Warehouse): SSD/HDD. Traag, maar permanent. Je `Program.cs` file staat hier totdat je hem start.

2. De Hartslag: Fetch-Decode-Execute

Je CPU doet eigenlijk maar √©√©n ding, miljarden keren per seconde:

  1. Fetch: Haal de volgende instructie op uit het RAM (op adres van de Instruction Pointer).
  2. Decode: Wat betekent 10110010? Ah, het betekent "Tel 5 op bij Register A".
  3. Execute: De ALU voert de som uit.
‚Ñπ üí° Inzicht:
C# code (text) wordt gecompileerd naar IL, en de JIT (Just-In-Time) compiler maakt daar machinecode van die precies past in deze cyclus.

3. Geheugenhi√´rarchie (Waarom je C# traag kan zijn)

Niet al het geheugen is gelijk. Toegangstijden verschillen enorm:

Locatie Grootte Snelheid (Analogie)
CPU Registers ~1 KB 1 sec (Je hersenen)
L1/L2 Cache MB's 10 sec (Op je bureau pakken)
RAM (Main Memory) GB's 2 min (Naar de bieb lopen)
SSD / Network TB's Dagen/Jaren (Een brief naar de maan sturen)

Waarom boeit dit?
Als jij in C# een List<Class> maakt, verspreid je data over het RAM. De CPU moet telkens wachten ("naar de bieb lopen"). Als je een Array<Struct> gebruikt, staat alles netjes achter elkaar en past het in de Cache ("op je bureau"). Dit kan je code 100x sneller maken.

üß† Denk-Oefening 0.1

Stel, je hebt een programma dat heel vaak data van Disk (SSD) leest.

Vraag: Waarom helpt een snellere CPU (meer GHz) hier bijna niet?

Toon Antwoord

Omdat de CPU (nanoseconden) miljarden cycli staat te wachten op de SSD (milliseconden). Dit heet I/O Bound zijn. De bottleneck is niet de rekenkracht, maar de data-toevoer. Oplossing module 9: Asynchronous programming.
